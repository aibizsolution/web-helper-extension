# 디자인 가이드

`웹 도우미` 사이드패널 UI는 기능마다 다른 모양을 만드는 방식이 아니라, 이미 검증된 패턴을 반복해서 쓰는 것을 기본 원칙으로 합니다.

## 핵심 철학
- 관련 기능은 `항목별 그룹`으로 묶는다.
- 바깥 카드 안에 또 카드, 그 안에 또 카드가 들어가는 중첩 섹션은 피한다.
- 새 기능을 추가할 때는 새로운 스타일을 만들기 전에 기존 패턴을 먼저 재사용한다.
- 같은 역할의 UI는 탭이 달라도 같은 모양과 같은 간격을 유지한다.

## 공통 원칙
- 섹션 타이틀은 가능하면 배경 위에 직접 둔다.
- 실제 내용은 한 단계 안쪽의 그룹 카드나 스택에서만 보여준다.
- 그룹 내부는 `row/list/card stack`으로 정리하고, 불필요한 wrapper를 늘리지 않는다.
- 설명 문구는 짧고 직접적으로 쓴다.
- 같은 뜻의 설명을 제목 아래와 항목 아래에 반복해서 넣지 않는다.
- 부가 설명은 본문보다 `툴팁`으로 우선 보낸다.
- 이 프로젝트는 `모바일 대응`이 아니라 `데스크톱 Chrome 사이드패널 폭 대응`을 기준으로 본다.
- 좁은 폭 대응이 필요해도 모바일용 별도 브레이크포인트를 습관적으로 추가하지 않는다.

## 탭 원칙
- 메인 탭은 현재 사이드바 패턴을 유지한다.
- 메인 탭 순서는 `핵심 작업 -> 보조 기능 -> 관리 -> 외부 액션` 흐름을 따른다.
- 번역 성격이 같은 기능은 메인 탭을 따로 쪼개지 않고 하나의 메인 탭 안 서브탭으로 묶는다.
- 현재 권장 순서는 `번역 / 페이지 진단 / 검색 / 도구 / 반복관리 / 설정 / 오류 센터 / 최신버전`이다.
- 번역 탭 내부 권장 서브탭 순서는 `사이트 / 텍스트 / 기록`이다.
- 서브탭은 반드시 기존 `translate-subtabs` / `translate-subtab` 패턴을 재사용한다.
- 새 서브탭을 만들 때 pill형, segmented형 등 별도 스타일을 새로 만들지 않는다.
- 활성 상태 표현은 `aria-selected="true"` 기준으로 맞춘다.

## 그룹/카드 원칙
- 한 화면 안에서 같은 성격의 그룹 카드는 동일한 radius, border, padding을 쓴다.
- 그룹 카드 안에는 다시 카드형 섹션을 넣지 않는다.
- 예외가 필요하면 시각적 이유보다 정보 구조 이유가 먼저 설명돼야 한다.
- 공용 그룹 클래스가 이미 있으면 새 이름을 만들지 않고 `card`, `action-row`, `inline-select`, `chip` 같은 기존 언어를 먼저 쓴다.

## 컨트롤 원칙
- 기간 선택처럼 보조 필터는 chip 스타일을 쓸 수 있다.
- chip은 탭이 아니라 필터이므로, 서브탭과 시각적으로 구분한다.
- 항목 실행에 직접 연결되는 설명은 가능한 한 `항목명 + 툴팁 + 실행 컨트롤`로 묶는다.
- 항목형 액션은 기본적으로 `한 줄`에서 읽히게 만들고, 좁은 폭에서도 먼저 줄바꿈 예외를 만들지 않는다.
- destructive action은 `기간/옵션 + 삭제`처럼 한 묶음으로 보여서 무엇이 즉시 실행되는지 바로 이해되게 만든다.
- destructive action은 기본 버튼보다 한 단계 더 조심스럽게 보이게 하고, 가능하면 2단계 확인을 둔다.

## 새 기능 추가 체크리스트
1. 기존에 같은 역할의 UI가 있는지 먼저 찾는다.
2. 있으면 같은 클래스/마크업 패턴을 재사용한다.
3. 없으면 최소 단위의 공통 패턴으로 만든다.
4. 한 기능 때문에 전혀 다른 디자인 언어를 만들지 않는다.
5. 큰 UI 구조 변경 시 `AGENTS.md`와 이 문서를 함께 갱신한다.
